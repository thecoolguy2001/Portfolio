May 2026

Reported Bug Fix Pass

A focused two-week sweep through every open user report. The Milkdrop drop and the Apple Music / Spotify integration had pulled in a wider audience, and the bug log had grown faster than the feature log. May was a stop-the-line moment to clear it.

Camera permission stuck

The camera-background Visual Mode style would sit on "requesting" forever if the user denied camera access in the browser then re-enabled it in OS settings. Fix: re-query navigator.permissions on visibilitychange and reset the cyan permission frame.

Safari recording cap

Recordings cut off at ~60s on Safari due to in-memory Blob chunk accumulation. Fix: stream chunks to disk via the File System Access API where available, and surface the cap as a 50s warning toast on Safari fallback.

FFT mid-band off-by-one

Effects in the mid-bandpass cluster (22–29) were reading the wrong FFT bin range, producing flat reactivity at low volume. One-line fix; visible difference on quiet vocal tracks.

Augment cycling on Pause (iOS)

The 300ms Augment timer kept firing while the canvas was paused on iOS. Now the timer is bound to the play state, not the render loop, so Pause actually pauses everything.

Silent custom-effect upload fail

Custom effects over 1MB silently failed to save. Now the C menu shows the file size next to the save button and rejects with a specific reason instead of doing nothing.

Overlay z-order

Pressing H while the Pro password sheet (Q) was open layered the help overlay underneath. Fixed the modal stack so the most recently opened overlay always wins, and Esc dismisses top-down.

Spotify token refresh

Long sessions (>1hr) lost Spotify playback when the access token expired mid-track. Now the refresh handshake runs in the background and the player rebinds without dropping the canvas.

Draw Mode undo race

Mashing Backspace fast erased strokes out of order because the stroke buffer wasn't sequenced. Replaced with a stable LIFO stack — undo always pops the last visible stroke.

Transition-aware effects

Track-change events from both SDKs fire effect crossfades so the visual lands cleanly on the downbeat of the next song.

Auth lives in a sheet

Sign-in opens in the same dark, rounded overlay pattern used elsewhere — the canvas dims, a single sheet asks for one thing, and dismisses on success.

Privacy-first scopes

One playback model, three providers

The free path stays whole

March 2026

Custom Effect Creator v2 — Hot Reload & Sharing

The July 2025 launch had shipped the four-function template and a save button. Less than a year of watching users in the editor surfaced two clear friction points: the iteration loop kept breaking the audio context, and there was no way to share a finished effect with a friend. v2 was the answer to both.

Hot reload

Save (⌘S) re-compiles the shader and rebinds the effect without dropping the audio context. The track keeps playing, the FFT keeps running, and the canvas just changes. No more reload-and-find-your-place.

Shareable links

One button encodes the effect (JS + GLSL + metadata) into a URL fragment. Send it to anyone running Pro and they import it with one click — no GitHub gist, no copy-paste, no account.

January 2026

Engine Refactor — One Render Path

Two years of additions had layered render paths on top of each other: the original native effects, the Milkdrop integration, custom user effects, the Visual Mode camera background layer, Game Mode overlays. Each had been bolted on around the previous one. January was the cleanup pass — collapsing all five into a single pipeline with one entry point and one effect interface.

One render loop

A single requestAnimationFrame entry point owns the frame budget. Audio analysis, effect update, camera ingest, and post-processing all run in a fixed order with measured slices.

Common effect interface

Native, Milkdrop, and custom effects all implement the same three-method contract: init(gl), update(audio, time), draw(). The renderer doesn't care which kind it's drawing.

~3,000 lines deleted

Six adapter layers, four duplicate FFT consumers, and three near-identical color-conversion utilities collapsed into one of each. The repo got materially smaller.

~12% perf gain

Quantum Architect and Wormhole — the two heaviest effects — measured 12–14% faster on the baseline laptop after the refactor. No perf work, just removing redundant passes.

Test surface tripled

One interface meant one test harness. Coverage went from "the audio analyser has unit tests" to "every effect can be smoke-tested headless against a synthetic FFT trace."

Foundation for what's next

Apple Music, Spotify, and Custom Effect Creator v2 all shipped on top of the new pipeline. None of those would have been pleasant to build against the old render path.

Refactor before, not during

The cleanest moment to flatten render paths was right before a year of integration work, not in the middle of it. Doing it first turned every later feature into an additive change instead of another bolt-on.

No user-visible regressions

The whole point of an internal refactor is that nobody outside the codebase notices. Side-by-side recordings on twenty representative tracks confirmed pixel-level parity before the old paths were removed.

August 2025

Performance & Browser Support

Fifty+ native effects plus the full Milkdrop preset library, real-time FFT, MediaRecorder, optional camera ingest, and a third-party shader sandbox — all targeting 60fps on hardware that ranges from a six-year-old MacBook to a brand-new iPhone. The constraints had to be designed in, not patched on.

60fps target

Frame budget: 16ms. Render loop, audio analysis, and shader uniforms all share that budget. Any effect that can't hold it on a baseline laptop ships behind the perf toggle.

WebGL 2.0 required

Shader-heavy effects assume modern context. Hard requirement, communicated up front rather than letting users hit silent black screens.

Codec fallback chain

VP9 25mbps → VP8 20mbps → H.264 (Safari). Recording works on every supported browser with the best available codec, automatically.

Photosensitivity gate at first launch

The strobe-capable effects exist. The launch overlay names that risk explicitly and waits for an "I Understand — Continue" press before the canvas activates — no silent autoplay.

Shared engine across surfaces

The Swift iOS / macOS ports re-implement the audio path natively but share the effect catalog and color system with the web build. Surface-specific optimizations live in the port, not in the effect.

Honest fallback for DRM tracks

Apple Music's DRM blocks AnalyserNode access on protected tracks. Rather than silently looking broken, the visualizer falls back to dummy analyzer data and surfaces the reason in the song corner.

Draw Mode (Pro)

The first creative-tooling addition after the launch effects. Instead of a single effect filling the screen, the cursor itself becomes a brush — every effect in the library can be painted as a stroke onto the canvas, layered with other effects, and erased one stroke at a time. Toggle with D, switch brush effect with arrow keys, and intensity 1-9 still applies to every drawn stroke at once.

Effect-as-brush

Click and drag spawns instances of the active effect along the cursor path. Each stroke remembers which brush drew it — switching effects mid-drawing layers cleanly without overwriting.

Fade Mode for performance

Single-stroke undo

Backspace erases the last stroke. No multi-step history, no panel — one key, one undo. Lets the user iterate fast without a UI surface getting in the way of the canvas.

One help sheet, every key

The H Controls overlay lists every binding the app responds to — including the Draw Mode keys (F, Backspace) — so a returning user can find any control without leaving the canvas.

June 2025

Visual Mode (Particle Style Cycling)

A single keystroke that re-skins the particle field without changing the active effect or its audio reactivity. Toggle with V and the rendering style of every particle on the canvas swaps — same motion, same beat response, different visual grammar.

Style as a parallel axis

Effect and mode were already the two cycling dimensions. Visual Mode adds a third — particle render style — so a user can keep the song, keep the effect, and still find a fresh look without breaking the moment.

Audio reactivity is preserved

Style switching only changes how each particle is drawn — not how it moves. Bass-driven scale, treble-driven color shift, and stroke timing all carry across styles unchanged.

One label, no menu

The active style appears as a single monospace label in the bottom-left — particle visual: style 1. No picker, no overlay; V advances to the next style, the label updates, and the canvas stays the canvas.

April 2025

Accessibility & Inclusive Design

An audio-reactive visualizer is, by nature, a hostile space for some users. That doesn't excuse the product from being thoughtful — it raises the bar. Every accessibility decision below was made before launch, not retrofitted.

Photosensitivity warning at first launch

Strobe-capable effects exist in the library. The warning overlay names that risk explicitly and requires acknowledgement before the canvas activates. No silent autoplay.

Intensity is user-controlled

Augment Mode and effect intensity both live behind opt-in keys — X and 1-9. Nothing auto-escalates without a user pressing for it, so the floor of motion stays where the user left it.

Keyboard-first interaction model

Every action has a key. No required mouse gesture, no required touch gesture. Users on assistive switches or keyboard-only setups have parity with mouse users.

State cues are never color-only

Recording renders as ● recording... in the same neutral gray as every other status label — the dot's there for shape, the word carries the meaning. Draw Mode and Visual Mode also label their state in words (draw mode: on, particle visual: style 1), so color-blind users get the same information density as everyone else.

Permission prompts get framed

Camera, microphone, and Apple Music permissions all surface inside the app with context before the browser-level prompt fires. Users opt in knowing why.

Honest error messaging

Apple Music DRM means some tracks can't be analyzed. The app says so plainly instead of looking broken. "Unsupported file" beats a frozen canvas, every time.

Captions on tutorials

The H help overlay text is high-contrast and uses system text scaling — readable at OS-level enlargement.

No timed dismissals

Help, password entry, and info screens stay open until the user dismisses them. No "30 seconds and you've missed it" patterns.

System pause respected

Tab-out, OS lock, and "do not disturb" cleanly pause analysis instead of hammering the GPU off-screen.

July 2025

Custom Effect Creator (Pro)

The biggest "give the user a hammer" feature in the app. Pro users can write their own visual effect from scratch using JavaScript and WebGL shaders, hit save, and have it run inside Visualizer alongside the built-in library. Press C to open the Custom Effect Menu — Load Effect or Create New Effect — and the four-function template is the only contract.

Four functions, no boilerplate

A custom effect is just create, animateInitial, animateReactive, and adjustIntensity. Only create is required — the rest are opt-in for animation and audio reactivity.

Audio data as arguments

Reactive function receives three positional arguments — bass, mid, treble at indices 0, 1, 2 — accessed via arguments[0..2]. All values 0-255. Same FFT data the built-in effects use, surfaced through the simplest possible signature.

ShaderMaterial-friendly

THREE.js, the scene, camera, renderer, and the audio context are all exposed as globals. Advanced users get full ShaderMaterial access; beginners can use MeshBasicMaterial and never touch GLSL.

Sharing via clipboard

Save an effect and the shareable code copies to the clipboard automatically and pins to the bottom-left for 60 seconds. Recipients paste it into Load Effect to run it locally — no account, no server.

May 2025

Recording & Export

From day one, listeners and DJs asked for the same thing: a way to keep what they made. Recording shipped as a single R press — no settings sheet, no codec picker, no upload step. Press once to start, press again to stop, file downloads to disk. The tradeoff: pick sane defaults and tune them quietly under the hood.

One key, no dialog

R starts and stops. While recording, an ● recording… pill sits in the canvas so the user is never unsure whether they're capturing.

Browser-aware codecs

VP9 at 25mbps where available, VP8 at 20mbps as fallback, H.264 on Safari. Container is WebM on Chrome / Edge / Firefox and MP4 on Safari. The user never picks — the browser decides what works.

60fps at 320kbps audio

Targeting 60fps so the recording matches what the user actually saw. Audio rides at 320kbps so vocals and high-end percussion survive the encode.

Spacebar-first init

Recording requires the audio context to exist. The first spacebar initializes it; without that, recording silently captures empty audio. Documented up front, not buried in troubleshooting.

March 2025

Engineering Architecture

The render loop has to do four things every 16ms: read audio, run an FFT, update the active effect's uniforms, and draw. Plus an optional camera ingest, plus an optional MediaRecorder branch, plus optional shader code being hot-loaded by a Pro user. Here's how the pipeline is wired.

Audio Source → AudioContext → AnalyserNode → FFT (1024)
↓
bass / mid / treble (0-255) → Effect.animateReactive() → Shader Uniforms
↓
THREE.Scene → WebGLRenderer → <canvas>
— optional branches —
getUserMedia(camera) → VideoTexture → Shader Uniforms
↕
<canvas>.captureStream → MediaRecorder (VP9 / VP8 / H.264) → Blob → download

Single requestAnimationFrame

One render loop owns the frame. Effect updates, audio reads, camera texture refresh, and game logic all dispatch from the same tick — no rogue timers, no drift.

Custom effects share globals

Pro user code runs against the same scene, camera, renderer, audioContext, and visualElements the built-in effects use. No sandbox tax, no second pipeline.

1024-sample FFT

Tuned for 60fps. Larger windows gave better frequency resolution but added latency the eye picked up; 1024 hit the sweet spot — fast enough to feel synchronized, dense enough to separate the bass band from the mid.

Native-port engine reuse

The Swift iOS / macOS ports re-implement the audio path natively but share the effect catalog and color system 1:1 — one product, three surfaces.

Mobile Gesture Map

Mobile parity wasn't an afterthought — it shipped at launch. Every desktop key has a touch equivalent, and the gesture grammar was designed so a returning user never has to look up a control.

⇡ swipe up — mode ⇣ swipe down — mode ⇠ effect effect ⇢
tap — play / pause
double-tap — intensity
triple-tap — next song
long press — help
pinch — visual mode
2-finger tap — info
2-finger hold — augment
2-finger swipe up — pro

Gesture zones — perpendicular axes (vertical = mode, horizontal = effect) with tap counts and finger counts modulating action class.

Perpendicular axes

Vertical swipe = playback mode. Horizontal swipe = effect. Same model as the keyboard arrows. Switching surfaces (web → iPhone) doesn't reset muscle memory.

Finger count modulates intent

One finger = primary action. Two fingers = power-user / pro action. Pinch = visual style cycling. Long press = help. Each gesture class is its own mental bucket.

Tap-count escalation

Tap → play/pause. Double → cycle intensity. Triple → next song. Four → mute. Each tap up the chain costs more deliberateness, and that maps to action consequence.

R = Record. G = Game. D = Draw. C = Custom (code). H = Help. I = Info. Q = unlocK (one of the only ones not literally letter-matched, but Q's lonely top-left position made it the obvious "secret" key).

February 2025

Pro Tier & Reactivity Controls

The expansion plan needed a way to fund itself and a way to keep the free experience uncluttered. The answer was a Pro tier — paid users got a password that unlocked effects 11-50, Game Mode, the Custom Effect Creator, Draw Mode, file uploads, and Apple Music — while everyone else still got a complete, never-watered-down free product with the original ten effects.

Password, not account

Pro unlocks via a password — desktop presses Q, mobile uses two-finger swipe up. No login, no email collection, no sync server. The product stays offline.

Free is full, not crippled

Effects 1-10 are the most polished, hand-built ones, not a teaser set. Pro is additive — more effects, more modes — never a feature removed from free.

Intensity 1-9 across everything

One global intensity scale (number keys, or double-tap on mobile) drives every effect — built-in, custom, or drawn. 1 is subtle, 9 is maximum visual impact. One mental model, every surface.

Augment Mode

Press X (or two-finger hold on mobile) to rapid-cycle visual variations every 300ms — synchronized to the music for moments that need to escalate hard.

Effect Shuffle

Press O to auto-switch effects as songs change. Built for set lists where the user wants surprise without taking their hands off the audio.

Visual Mode Styles

Keep pressing V to cycle particle render styles — Style 1, Style 2, Style 3, and so on. Same effect, same audio reactivity, different visual grammar each cycle.

Every binding in one sheet

Press H for the Controls overlay — a single dark rounded sheet that lists every key the app responds to, from Spacebar through the pro keys (C / D / G / X) and back to H itself. One reference, always reachable, no second tab.

February 2025

Visual Design Direction

The visual language had one job: get out of the way of the music. Every choice — color, type, surface — was made to push the visualization forward and let the chrome recede.

Pure black canvas

Not "near-black" — actual black. Saturated effect colors read as emissive, and OLED screens get genuine black pixels instead of a dark gray field.

Effects own their palette

Each effect picks its own colors — Particles is a rainbow scatter, Aurora is greens and purples, TempleOS is its own thing. The app provides no global accent because the visualization is the color story.

Chromeless by default

No header, no buttons, no playback bar. The only persistent elements are two tiny corner labels — mode and song. Everything else is summoned with a key and disappears when you're done.

Monospace for status

Status text is monospace and low-contrast — different family from anything an effect could draw, so it always reads as "the app talking" rather than "part of the visual."

Method

Two perpendicular axes

Mode lives vertically (arrow up/down, swipe up/down). Effect lives horizontally (arrow left/right, swipe left/right). Together they map the entire main canvas with two gestures.

Pro is additive, not nested

Pro features sit beside free ones in the IA — same level, just gated. The user never has to "go into Pro" to use a feature. They press the key; if locked, the password sheet asks for unlock.

Modes vs. Actions

Toggleable modes (Game / Draw / Visual) are state. Discrete actions (Record / Info / Help) are events. The IA keeps them in different branches so they read differently in the help overlay too.

December 2024

Five Playback Modes

The audio source had to scale from "I just want a track to play" to "I'm running this through a hardware synth at a show." That meant five distinct input modes, all switchable from the same axis — arrow up/down on desktop, vertical swipe on mobile — so the user never has to leave the canvas to change source.

Mode 1

Playlist

Default. Curated built-in tracks ready to play with zero setup. N / B step through tracks (or triple-tap on mobile).

Mode 2 — Pro

File Select

Drag-and-drop or file picker for MP3, WAV, M4A, FLAC. Press P to add files into a custom playlist; N / B walk it.

Mode 3

Microphone

Live audio in. Designed for performances, jam sessions, and synth players. Browser permission flow is framed inside the app, not silent.

Mode 4 — Pro

Apple Music

Authorize via MusicKit JS, then any track in the user's library plays through the visualizer. DRM caveats are surfaced honestly — some tracks fall back to dummy analyzer data.

Mode 5 — Pro

Spotify

Authorize via the Web Playback SDK, then play full tracks with metadata, playback control, and transition hooks the visualizer can time effects to. Premium-tier accounts only — the SDK requirement.

One axis, five modes

Mode lives on the vertical axis; effect style lives on the horizontal axis. Two perpendicular gestures cover every navigation action without a menu.

Permission as a moment

Microphone and Apple Music both ask for authorization through framed in-app screens, not bare browser prompts. The user understands what's being granted before they grant it.

Honest DRM messaging

Apple Music's DRM means some tracks can't be analyzed — the visualizer falls back to dummy data and tells the user, instead of silently looking broken.

First Launch & Onboarding

The first thirty seconds had to do four things at once: warn about photosensitivity, prove the canvas was alive, get permission to play audio, and not feel like a setup screen. The solution was three sequential states with no menus and no buttons that don't matter.

Step 1 — Warning

⚠️ Flash Warning
If you are sensitive to flashing lights, please use caution.
I Understand — Continue

Acknowledged before any motion ever runs. No autoplay, no exception.

Step 2 — Spacebar Prompt

Press Spacebar to Start
mode: playlist

Single-key entry. The spacebar press doubles as the audio context init that recording needs later.

Step 3 — Active Canvas

[ CANVAS ]
mode: playlist song: nirvana

Final state — chromeless. The user is now in the product. Total time: about three seconds.

Photosensitivity is a first-class concern

The warning is the first thing on screen, not buried in settings. Acknowledging it is the only required gate — no email, no signup, no consent dialog stack.

Spacebar does double duty

"Press Spacebar to Start" gets the user into the product AND initializes the AudioContext (browsers gate audio behind a user gesture). The same key later starts and stops playback — one action, three jobs.

Mobile parity

On mobile the prompt becomes "Tap to Start" and the same gesture initializes audio. The flow doesn't fork — same three steps, different gesture grammar.
